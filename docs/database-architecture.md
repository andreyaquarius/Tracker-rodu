# Майбутня архітектура даних

## Мета

Перенести робочі дані «Трекера Роду» з одного JSON-файлу на Google Drive до PostgreSQL у Supabase, не втративши наявні записи, конструктор розділів, прикріплені файли та резервні копії.

Поточний застосунок поки не перемикається. SQL у каталозі `supabase/migrations` є підготовленою основою для окремого тестового середовища.

## Основні складові

- **Supabase Auth + Google**: вхід користувачів через Google.
- **PostgreSQL**: проєкти, записи, зв’язки, ролі та налаштування.
- **Supabase Storage**: приватні фото, аудіо, скани й документи.
- **Google Drive**: використовується лише як одноразове джерело імпорту старої бази.

## Проєкти та доступ

Кожне дослідження верхнього рівня стає окремим проєктом. Користувач може належати до кількох проєктів.

| Роль | Можливості |
| --- | --- |
| Власник | Керує проєктом, учасниками, структурою розділів і всіма записами |
| Редактор | Створює та редагує записи й файли |
| Переглядач | Лише переглядає дані та файли |

Власник запрошує учасника за електронною адресою та вибирає роль. Запрошення має строк дії. Після входу через відповідний Google-акаунт користувач приймає запрошення.

## Захист даних

У всіх таблицях проєкту є `project_id`. Політики Row Level Security перевіряють членство та роль на стороні бази даних. Тому приховування кнопок у застосунку не є єдиним захистом.

Файли зберігаються у приватному контейнері `project-attachments`. Шлях починається з ідентифікатора проєкту:

```text
<project-id>/<record-type>/<record-id>/<file-id>-<file-name>
```

Доступ до файлу перевіряється тими самими ролями, що й доступ до записів.

## Відповідність поточним даним

Стандартні масиви поточного JSON отримують окремі таблиці:

- `researches`
- `documents`
- `year_matrix`
- `tasks`
- `findings`
- `hypotheses`
- `persons`
- `person_relations`
- `archive_requests`

Конструктор зберігається у:

- `custom_sections`
- `custom_section_fields`
- `custom_records`
- `record_links`

Додаткові поля стандартних розділів описуються в `custom_field_definitions`, а їх значення залишаються у `custom_fields` відповідного запису у форматі JSONB.

Неповні та приблизні генеалогічні дати навмисно зберігаються як текст. Це не дає втратити значення на кшталт «близько 1850», «до 1864» або неповні дати.

## Межа відповідальності сервісів

Наявний `storageService` працює з цілим файлом бази та резервними копіями. Для PostgreSQL потрібен окремий рівень доступу до записів:

```text
Інтерфейс застосунку
        |
        v
dataService / repositories
        |
        +-- postgresDataProvider
        |
        +-- legacyJsonDataProvider (тимчасово для міграції)

backupService
        |
        +-- Supabase Storage

legacyImportService
        |
        +-- googleDriveStorageProvider
```

Не варто маскувати PostgreSQL під поточний файловий `StorageProvider`: це ускладнить пошук, спільне редагування та часткове завантаження даних.

## Правила першої версії

- Структуру проєкту та конструктор змінює власник.
- Редактор працює із записами й файлами.
- Переглядач не виконує операцій запису.
- Видалення власника зі складу проєкту заборонене.
- Службовий ключ Supabase ніколи не додається до браузерного застосунку.
- Імпорт виконується на сервері або одноразовим локальним інструментом зі службовим ключем.
