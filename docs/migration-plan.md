# План переходу на PostgreSQL

## 1. Підготовка

1. Створити окремий тестовий проєкт Supabase.
2. Зберегти повну ручну резервну копію поточного JSON та всіх файлів Google Drive.
3. Застосувати міграції з каталогу `supabase/migrations`.
4. Увімкнути Google як постачальника входу в Supabase Auth.
5. Додати локальну та майбутню робочу адреси застосунку до дозволених адрес перенаправлення.

На цьому етапі чинна версія «Трекера Роду» продовжує працювати як раніше.

## 2. Новий рівень даних

Створити в застосунку `dataService`, який працюватиме не з цілим JSON-файлом, а з окремими сутностями:

- проєкти й учасники;
- стандартні модулі;
- конструктор розділів;
- прикріплені файли;
- пошук;
- журнал змін.

Першим новим постачальником буде `postgresDataProvider`. Поточний JSON-постачальник тимчасово залишиться як джерело імпорту та аварійного відновлення.

## 3. Імпорт наявного проєкту

1. Користувач входить через Google.
2. Створюється профіль і перший проєкт, де цей користувач є власником.
3. Поточні масиви JSON імпортуються в таблиці зі збереженням старих ідентифікаторів, якщо вони є UUID.
4. Для не-UUID ідентифікаторів створюється таблиця відповідності старого та нового ID.
5. Спочатку імпортуються батьківські записи, потім зв’язки.
6. Файли копіюються з Google Drive до приватного Supabase Storage.
7. У таблиці `attachments` записуються нові шляхи файлів.

Рекомендований порядок:

```text
profiles
projects
researches
persons
documents
year_matrix
tasks
findings
hypotheses
archive_requests
custom_field_definitions
custom_sections
custom_section_fields
custom_records
relations and links
attachments
```

## 4. Перевірка імпорту

До перемикання потрібно автоматично порівняти:

- кількість записів кожного типу;
- назви та основні поля записів;
- кількість зв’язків;
- кількість, розмір і контрольні суми файлів;
- кастомні поля та записи конструктора;
- відкриття випадково вибраних файлів;
- пошук за кількома відомими прізвищами, місцями й роками.

Імпорт вважається успішним лише за відсутності невідомих втрат. Помилки записуються в окремий звіт, а поточний JSON не змінюється.

## 5. Перемикання

1. У локальній тестовій версії ввімкнути PostgreSQL через змінну середовища.
2. Перевірити створення, редагування, видалення, файли та права трьох ролей.
3. Провести повторний чистий імпорт з актуальної резервної копії.
4. На короткий час заборонити зміни у старій версії.
5. Перемкнути робочу версію на PostgreSQL.
6. Зберегти Google Drive як резервне сховище та шлях швидкого відновлення.

Подвійний запис одночасно в JSON і PostgreSQL у першій міграції не рекомендований: він створює ризик розходження двох копій.

## 6. Обов’язкові перевірки прав

- Власник бачить і редагує свій проєкт.
- Редактор змінює записи, але не керує учасниками.
- Переглядач не може записати дані навіть прямим запитом.
- Сторонній користувач не бачить проєкт, записи або файли.
- Запрошення приймається лише Google-акаунтом із вказаною адресою.
- Прострочене або відкликане запрошення не працює.
- Файл одного проєкту недоступний учасникам іншого.

## 7. Відкат

До підтвердження стабільної роботи:

- не видаляти старі файли з Google Drive;
- зберігати останній JSON-експорт;
- не змінювати формат резервних копій;
- мати можливість тимчасово повернути локальний режим лише для читання.
